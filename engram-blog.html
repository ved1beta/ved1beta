Engram: Conditional Memory via Scalable Lookup

DeepSeek’s answer to the second sparsity problem — the one MoE didn’t touch.
by Ved · systems engineer at Axolotl AI
github / ved1beta · x / @ant_vedaya

DeepSeek papers are some of the most structurally interesting pieces of modern ML literature — they tend to identify a flaw the rest of the field has accepted as background noise, then engineer the minimal fix for exactly that flaw. Engram is one of those papers.

Modern transformers face a stubborn bottleneck as they scale: memory. Not GPU memory — the model’s ability to retrieve what it knows. The mechanism we have, attention plus feed-forward layers, has no native primitive for knowledge lookup. To recall a static fact, the model has to simulate retrieval through computation. Engram replaces that simulation with a real lookup table.

“Transformers lack a native primitive for knowledge lookup, forcing them to inefficiently simulate retrieval through computation.” — Engram paper, abstract
· · ·

§ 01Two sparsity problems, one solved

Beneath the standard transformer recipe sit two distinct sparsity problems. Mixture-of-Experts addressed one of them and became the de facto frontier-model architecture. The other was left structurally unaddressed.

Compute sparsity — solved by MoE

At any moment, only a fraction of model parameters need to fire. MoE said “not every expert needs to activate for every token,” routed tokens to the relevant experts, and left the rest idle. Problem one closed.

Memory sparsity — left open

At any moment, only a fraction of stored knowledge is actually retrieved. But that knowledge is buried in weight matrices. To recall “Alexander the Great,” the model spends multiple early layers reconstructing what could have been a single lookup.

Engram is the answer to the second problem. The diagnosis is precise: static knowledge is being reconstructed through dynamic computation. The fix is also precise: a sharded, deterministic, differentiable lookup table that sits in front of attention.

§ 02Before Engram — what an N-gram is

The lookup keys in Engram are N-grams. An N-gram is just a contiguous sequence of N tokens — instead of looking at one token in isolation, you look at pairs, triples, or longer windows together.

The insight is plain. “Alexander” alone is ambiguous. “Alexander the Great” is a fixed, recurring concept — and any model that sees this trigram should retrieve the same knowledge every time, regardless of context. That is exactly what a lookup table does. Hash the N-gram to an index, retrieve the stored embedding, inject it into the hidden state. No attention. No multi-layer reconstruction. O(1).

Why not just grow the FFN? Resolving a multi-token entity through pure computation consumes early attention and FFN depth — sequential capacity spent on a static reconstruction. Engram offloads that work to a direct lookup, freeing depth for genuine reasoning.

§ 03The Engram architecture

Engram sits before both attention and MoE inside selected transformer layers. When a token sequence arrives, the module runs two phases: retrieval and fusion.

Sparse retrieval via hashed N-grams

Before lookup, raw token IDs pass through a tokenizer compression step. Subword tokenizers assign different IDs to semantically equivalent forms — "Apple" vs "apple", leading-space variants, and so on. The compression projects all of these onto canonical IDs, achieving roughly a 23% reduction in effective vocabulary. Fewer distinct N-gram keys means denser, better-trained embedding rows.

With compressed IDs in hand, the module computes N-gram hashes using multi-head hashing. For each N-gram order (bigram, trigram) and each of K heads, a deterministic multiplicative-XOR function maps the context to an index inside a prime-sized embedding table:

zt,n,k = φn,k(gt,n),    et,n,k = En,k[ zt,n,k ]

Each head uses a unique prime modulus as its table size. This is the key design call: if two heads shared a modulus, the same N-gram would always hash to the same bucket across them and the heads would be redundant. Distinct primes give statistically independent hash spaces. The retrieved memory is the concatenation across orders and heads:

et = concatn=2..N, k=1..K et,n,k

Context-aware gating

The retrieved embedding et is a static, context-independent prior. It captures what the model knows about this N-gram in general — not whether that knowledge is relevant here. A context-aware gate resolves that.

The hidden state acts as a dynamic Query (it has already passed through earlier attention layers and carries global context). The retrieved memory acts as the source of both Key and Value. A scalar gate is computed as:

αt = σ( RMSNorm(ht)⊤ RMSNorm(kt) / √d )

The gated output is ṽt = αt · vt. If retrieved memory contradicts the current context, the gate decays toward zero and the lookup is suppressed. If it aligns, it passes through at full strength. Semantic alignment, enforced at the module level.

Finally, a short dilated depthwise causal convolution refines the gated output. With kernel size 4 and dilation set to the max N-gram order, it sees positions t, t−3, t−6, t−9 — capturing repeating sub-word patterns spaced at N-gram boundaries. Critically, the conv is zero-initialised: Engram contributes nothing at initialisation, which makes it safe to drop into a pretrained model without disturbing existing representations. Training gradually switches it on.

§ 04System implementation

A 130k-token vocabulary with bigrams and trigrams yields a table with hundreds of millions to billions of rows. It cannot live on a single GPU. The paper’s solution is to shard the table across all available GPUs and use All-to-All communication to route each lookup to the device that owns it.

The All-to-All protocol runs in three rounds. First, each rank tells every other rank how many IDs it is sending. Second, the IDs are scattered — rank r sends each ID to the rank that owns the corresponding shard. Third, each rank does a local lookup, then scatters the resulting vectors back to the requesters. A sort-by-owner-rank and subsequent unsort keep the output in original query order, fully differentiable.

Training vs inference asymmetry. During training the table lives on GPU memory, split across ranks. During inference, the entire table is offloaded to CPU host memory. Because Engram’s hashing is deterministic, the system knows exactly which rows the next batch will need before the forward pass starts — so the host prefetches in parallel with compute and hides the transfer entirely.

§ 05Multi-branch integration

The DeepSeek backbone is not a single-stream transformer. It uses a multi-branch (mHC) architecture where the residual stream expands into M parallel branches. Engram adapts via a parameter-sharing strategy: one shared Value projection WV across all M branches, but M distinct Key projections {WK(m)} for branch-specific gating. Each branch gates the same retrieved memory vector independently, enabling fine-grained, branch-aware injection.

This also lets the linear projections fuse into a single dense FP8 matmul — one WV stacked with M WK — maximising GPU utilisation with negligible overhead.

§ 06Distributed training

The full model trains with DDP wrapping the entire network, but the optimiser is split into two groups, because embeddings and the backbone have fundamentally different gradient structures.

Backbone parameters receive dense gradients: every parameter is updated on every step. They benefit from weight decay and standard AdamW at lr = 4e-4.

Embedding parameters receive sparse gradients: only the rows touched by tokens in the current batch get updated. Most rows see zero gradient on any given step. Weight decay on sparse embeddings is actively harmful — unused rows would decay toward zero even though they were simply never queried. The fix is plain Adam with no weight decay, and a 5× higher learning rate (lr = 20e-4) to compensate for infrequent updates.

  1. DistributedSampler assigns disjoint token batches to each rank — every rank sees different data.

  2. Forward pass. DDP runs in sync; inside the Engram module, All-to-All fetches embeddings across shards.

  3. Backward pass. DDP all-reduces backbone gradients across ranks. Embedding gradients stay local — each rank only accumulated gradients for the rows it owns.

  4. Optimiser step. AdamW updates the backbone, plain Adam updates embeddings. Independent groups, no interference.

§ 08Why this matters

MoE solved compute sparsity. Engram solves memory sparsity. Together, they address both structural inefficiencies of the vanilla transformer — not by scaling the same thing harder, but by recognising that the two tasks inside a language model (compositional reasoning and knowledge retrieval) have fundamentally different computational profiles and deserve different primitives.

What makes the paper worth reading carefully is the precision of the diagnosis. There is no vague “augmented memory” rhetoric. The flaw is named — static knowledge reconstructed through dynamic computation — and the fix is engineered for exactly that flaw. A lookup table. Sharded. Deterministic. O(1). Differentiable.

“We envision conditional memory as an indispensable modeling primitive for next-generation sparse models.” — Engram paper
